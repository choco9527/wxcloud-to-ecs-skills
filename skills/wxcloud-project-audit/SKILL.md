# 微信云开发项目审计

## 目标

在任何迁移动作开始前，先完成现状盘点，输出后续架构设计和迁移执行所需的最小事实集合。

这个 skill 解决的是“先搞清楚项目到底依赖了哪些平台能力，再决定怎么迁”。

## 何时使用

当用户出现以下意图时，优先使用本 skill：

- 先分析当前云开发项目
- 先盘点云函数、云数据库、云存储使用面
- 先确认迁移范围和优先级
- 先做迁移方案输入，不直接改代码
- 先识别哪些能力耦合最深、风险最高

如果用户已经明确进入“开始改后端/改前端/做切流”的阶段，再切到其他迁移 skill。

## 审计范围

至少覆盖以下六类对象：

1. 云函数
- 入口函数
- 聚合分发函数
- 定时触发函数
- 管理端或脚本触发函数

2. 前端调用入口
- 统一请求封装
- 分散调用点
- 环境判断逻辑
- 登录触发点

3. 数据存储
- 云数据库集合
- 关键数据模型
- 主键习惯
- raw/结构化并存情况
- 历史兼容字段

4. 文件存储
- 上传入口
- fileID 保存位置
- 回显位置
- 聚合文件/缓存文件/导出文件
- 历史文件兼容读取

5. 后台任务
- 定时触发器
- 手动触发任务
- 抓取、聚合、刷新、清理类任务
- 日志记录与失败重试方式

6. 身份与权限
- 微信登录态来源
- openid/unionid 使用方式
- token/session 策略
- 管理员权限判断
- 本地调试回退路径

## 工作流

1. 先找前端调用入口，不要一上来只扫云函数目录。
2. 列出所有云函数及其调用来源，区分主链路与后台链路。
3. 识别云数据库使用面，重点看哪些数据已经平台耦合严重。
4. 识别云存储语义，确认哪些地方依赖 fileID 而不是普通 URL。
5. 识别定时任务和后台任务，确认它们是否依赖云触发器。
6. 识别登录、鉴权、管理员权限与本地调试路径。
7. 汇总为迁移优先级、风险点、建议顺序和待验证清单。

## 常见遗漏点

审计时重点防止漏掉以下内容：

- 前端某些页面直接调用 `wx.cloud`，没有走统一封装
- 云函数里混合了多个职责：读、写、任务、管理操作
- 历史字段或 raw 数据承担真实业务事实源
- 上传成功后使用的不是文件本身，而是 fileID 或聚合文件地址
- 定时任务之外还有手动管理触发入口
- 管理员判断不只在后端，也散落在前端页面逻辑中
- 本地调试依赖临时 token、测试 openid 或特殊环境变量

## 输出模板

输出时统一使用以下结构：

### 1. 当前使用面摘要
- 云函数数量与主要分组
- 前端调用入口数量与形态
- 数据存储使用摘要
- 文件存储使用摘要
- 后台任务使用摘要
- 鉴权与权限摘要

### 2. 平台耦合点
- 强依赖云函数的链路
- 强依赖云数据库语义的链路
- 强依赖 fileID/云存储语义的链路
- 强依赖云触发器的链路
- 强依赖微信身份态的链路

### 3. 迁移优先级
- P0：主链路必须先迁的对象
- P1：影响联调和灰度的对象
- P2：可以后置的对象

### 4. 高风险点
- 结构兼容风险
- 文件访问风险
- 定时任务风险
- 权限与鉴权风险
- 环境切换风险

### 5. 建议迁移顺序
按阶段给出，不少于三步：
- 盘点完成后先迁什么
- 哪些能力第二批迁
- 哪些能力最后收口

### 6. 待验证清单
- 需要本地联调验证的链路
- 需要测试环境验证的链路
- 需要灰度阶段重点观察的链路

## 注意事项

- 不要直接进入代码改造；先盘点完成再设计。
- 不要只统计数量，要识别主链路、后台链路和管理链路。
- 不要把迁移期兼容壳默认当成长期方案。
- 审计输出必须可供后续架构设计直接使用，不能只给模糊结论。
