# 通用迁移原则

## 1. 先做能力盘点，不要直接改代码

迁移开始前，先识别项目实际使用了哪些平台能力：

- 云函数
- 云数据库
- 云存储
- 定时触发器
- 登录与用户身份链路
- 前端环境判断与请求入口

没有盘点就开始改造，容易漏掉平台耦合点。

## 2. 前端调用契约优先稳定

迁移期优先保持前端调用方式稳定，避免同时大规模改动前后端。

常见做法包括：

- 保留原有请求入口的语义
- 兼容旧入参结构
- 兼容旧返回结构
- 将环境切换逻辑收口到统一请求层

## 3. 迁移期允许兼容壳，但必须标记边界

兼容层可以存在，但必须明确是迁移期策略，不应扩散到新功能。

兼容层示例：

- raw 数据兜底读取
- 旧字段与新字段并行
- 历史 fileID 与新 URL 并行兼容
- 临时开发登录 token

## 4. 本地调试与正式环境必须分离

本地调试可以使用专用策略，但不得污染正式环境：

- 本地专用 API 域名或端口
- 本地调试 token
- 本地文件 URL 改写
- 本地数据库连接方式

正式环境必须满足：

- 真实微信登录链路
- HTTPS
- 合法业务域名
- 明确的日志、告警与回滚方案

## 5. 文件链路和后台任务通常比 API 更容易出问题

迁移时不要只盯 HTTP 接口，还要优先检查：

- 上传与访问 URL
- 历史文件兼容
- 定时任务触发方式
- 任务日志与失败重试
- 聚合文件或缓存文件的生成方式

## 6. 切流必须分阶段进行

推荐顺序：

1. 本地联调通过
2. 测试或体验环境联调通过
3. 核心链路灰度切换
4. 观察稳定性后再扩展范围
5. 明确保留旧链路的短期回滚能力

## 7. 架构设计要从“替换平台能力”出发

迁移设计时，应把原平台能力逐项映射到目标架构：

- 云函数 -> 后端服务/API
- 云数据库 -> 自建数据库
- 云存储 -> 文件服务/对象存储
- 云触发器 -> cron/systemd timer/任务调度器
- 云身份 -> 微信登录 + 自建鉴权
